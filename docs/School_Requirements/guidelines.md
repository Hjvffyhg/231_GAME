# Final Game Output Guidelines

## 1. Groupings and Collaboration
* Each group must have a complete list of members.
* Each member must have an assigned role or task.
* The group must submit only one final output.
* All members are expected to contribute to the development, documentation, and presentation.
* Non-participating members may be reported to the instructor for evaluation.
* The group is responsible for checking the completeness of the final submission.
* Any changes in group members must be approved by the instructor before submission.

### b. CPU Scheduling Algorithms
Each group is required to incorporate two assigned CPU scheduling algorithms into their final game output. The selected algorithms must be clearly shown in the game mechanics, system behavior, task processing, enemy actions, level progression, or player interaction design.

| CPU Scheduling Algorithm | Description |
| :--- | :--- |
| Shortest Job First | Prioritizes the process with the shortest burst time. |
| Shortest Remaining Time First | Executes the process with the shortest remaining time and allows preemption. |
| Longest Job First | Prioritizes the process with the longest burst time. |
| Longest Remaining Time First | Executes the process with the longest remaining time and allows preemption. |
| Round Robin | Assigns each process a fixed time quantum. |
| Highest Response Ratio Next | Selects the process with the highest response ratio based on waiting time and burst time. |
| Priority Preemptive | Executes the highest-priority process and allows interruption when a higher-priority process arrives. |
| Priority Non-Preemptive | Executes the highest-priority process without interruption once it starts. |

### c. Note on FCFS
* First-Come, First-Served (FCFS) is not included in the assigned CPU scheduling algorithms for the final game output because it was already given as the basic and foundational example.
* FCFS is considered the most fundamental CPU scheduling algorithm; therefore, it should already be understood by the students before applying more advanced scheduling concepts.
* For this progress report and final game project, each group must incorporate two assigned CPU scheduling algorithms from the given list, including FCFS.
* The focus should be on demonstrating how more advanced scheduling algorithms can be creatively applied to game mechanics, UI behavior, task prioritization, enemy movement, quest processing, or system interactions.

## 3. Allowed Development Tools and Content Guidelines
Groups may use different tools and approaches to develop, simulate, or present their final game output. The following are possible tools or methods that may be used:

| Tool / Method | Description |
| :--- | :--- |
| Game Engine | Tools such as Unity, Unreal Engine, Godot, or other game development platforms may be used to build the game. |
| Emulators | Emulators may be used if the game is designed or tested for a specific platform or environment. |
| Simulators | Simulators may be used to demonstrate game logic, CPU scheduling behavior, task flow, or system interaction. |
| Vibe Coding | AI-assisted or prompt-based coding may be used as support in developing game features, UI/UX designs, mechanics, or prototypes. However, the group must still understand, explain, and defend the output. |

## 4. Content Restrictions
The final game output, UI/UX design, poster, and demonstration must not contain inappropriate or offensive content. The following are strictly not allowed:

| Prohibited Content | Description |
| :--- | :--- |
| Sexually Suggestive Content | Any visual, dialogue, animation, character design, or theme that implies sexual content is not allowed. |
| Gender Discriminatory Content | Any content that promotes bias, stereotyping, insult, or unfair treatment based on gender is not allowed. |
| Profanity or Derogatory Language | Offensive, insulting, vulgar, or degrading words, phrases, symbols, or gestures are not allowed. |

All game outputs must be appropriate for an academic setting and must follow professional, respectful, and inclusive standards.

## 6. Grading
The Major Output is equivalent to 40% of the Final Grade. This means that the final game project, including all progress reports, demo, poster, and peer evaluation, will be part of the students’ major output grade.

Within this 40% Major Output, the project will be evaluated using a Project equivalent of 20%.

| Criteria | Description | Weight | Excellent | Good | Fair | Poor |

| Progress Report 2 | Development progress, implementation updates, diagrams, partial system/game functionality, and testing evidence | 10% | 9–10 pts | 7–8 pts | 5–6 pts | 0–4 pts |

## 7. Deductions
The approved Progress Report #1 – Ideaboard must be followed as the foundation of the final game output. Any unauthorized changes in the game title, genre, theme, main concept, core mechanics, assigned CPU scheduling algorithms, or UI/UX direction may result in deductions.

Groups must ensure that Progress Report #3 – UI/UX, the final game demo, and the infographic poster remain aligned with the approved concept from Progress Report #1.

Any revision from the approved Progress Report #1 must be properly communicated and approved before implementation. Unapproved changes may affect the group’s final score.

| Violation / Change Made | Deduction (Final Score) |
| :--- | :--- |
| Changing the approved game title without approval | -5 points |
| Changing the main game concept from Progress Report #1 | -10 points |
| Changing the assigned CPU scheduling algorithms without approval | -30 points |
| Submitting a UI/UX design that does not match the approved Progress Report #1 concept | -20 points |
| Presenting a final game output that is different from the approved Progress Report #1 concept | -20 points |
| Failure to explain or justify approved changes in the documentation | -15 points |
| Major inconsistency between Progress Report #1, Progress Report #3, game demo, and infographic poster | -20 points |
